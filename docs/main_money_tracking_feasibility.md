# 主力资金追踪数据基线审计报告 (Phase 5.1)

在进入第五阶段核心代码开发前，针对当前系统 `ai-trader` 的底层数据获取通道 (QMT) 与算力架构，进行了严格的主力资金数据可行性审计。

## 1. 可用字段清单 (QMT `get_full_tick` L1 基础数据)

通过扫描 `feeds/qmt_market_provider.py`、`core/fusion_engine.py` 及 `core/market_scanner.py`，确认系统当前依赖的 `xtdata.get_full_tick` 能瞬间稳定提取的字段包括：

- **基础量价**：`lastPrice`, `preClose`, `open`, `high`, `low`
- **累计量能**：`volume` (总成交量), `amount` (总成交额)
- **衍生指标**：`turnoverRatio` (换手率), `volRatio` (量比)
- **盘口挂单 (5档)**：
  - `bidPrice` [5档], `askPrice` [5档]
  - `bidVol` [5档], `askVol` [5档]

## 2. 不可用字段清单 (缺乏 L2 权限时无法直接获取)

由于绝大多数 QMT 默认账户不附带 Level 2 行情，以下字段**无法直接调用 API 现成读取**：

- ❌ **tick-by-tick 逐笔成交记录**：无法精确知道每一笔交易的准确单号。
- ❌ **买卖队列 (Order Queue)**：无法得知挂单是由少数大单构成还是大量散单构成。
- ❌ **现成的主动买/主动卖 (内外盘)**：QMT 不直接返回。
- ❌ **现成的大单净流入金额**：QMT 不直接返回。

## 3. 主力资金追踪可行性判断

### 🟢 级别 1：可直接实现 (利用现有字段)
- **委比 / 委差 / 挂单失衡 (Order Book Imbalance)**：
  *实现难度：极低。*
  在 `fusion_engine.py` 中已有雏形（利用 `sum(bidVol)` 与 `sum(askVol)`），可立刻提炼为主力托单/压单监控指标。

### 🟡 级别 2：需要间接估算 (状态机与时间序列差值)
- **主动买入 / 主动卖出 (内外盘估算)**：
  *实现难度：中。*
  **实现逻辑**：建立缓存状态，每 3 秒记录一次 Tick 快照，计算 `delta_volume = 当前volume - 上次volume`。若 `lastPrice >= askPrice[0]`，则将这段差量计入“主动买入（外盘）”；反之计入“内盘”。
- **大单净流入追踪 (Large Order Net Flow)**：
  *实现难度：中偏高。*
  **实现逻辑**：基于上述差值，如果 `delta_volume * lastPrice` 超过某阈值（如 100 万人民币），则判定该 3 秒内的成交为主力大单行为，累加进入 `main_money_net_inflow` 变量。

### 🔴 级别 3：当前无法实现
- **精准拆单识别**：无法识别主力利用冰山算法 (Iceberg) 将 1000 手拆分成 100 个 10 手的隐蔽吸筹动作。

## 4. 性能风险与推荐实现路线

### ⚠️ 性能风险预警 (OOM与CPU过载)
若按照传统的“全市场 5000 只股票”执行上述级别 2 的估算（每 3 秒做 5000 次差值计算和内存存储），将带来**毁灭性的性能灾难**：
1. 每秒高达 1600+ 次的数据比对会锁死 Python 的 GIL。
2. 将影响甚至卡死当前 Market Scanner 与 Dashboard 的正常运作。

### 💡 推荐实现路线 (精准狙击策略)

坚决放弃全市场 5000 股的实时追踪，**引入“漏斗继承制”**：

1. **追踪对象隔离**：新建 `core/tape_reader.py` (盘口阅读器)。
2. **标的降维**：它**只读取** `data_cache/market_candidates.json` 中排名前 30（或 Top 10）的股票。
3. **实时累加**：只对这 30 只股票维持 3 秒级别的 `get_full_tick` 轮询订阅，利用字典缓存上一刻的 `volume` 和 `amount`。
4. **生成大单画像**：输出诸如“资金净流入”、“大单发生频率”、“盘口托单失衡比”等指标至 `data_cache/main_money_tracking.json`。
5. **UI / Copilot 展示**：只作为辅助观测的展示层数据，绝不产生买卖信号。

---
> [!IMPORTANT]
> 承诺说明：本可行性报告规划的功能，完全遵守“不修改交易执行逻辑”、“不调用下单能力”、“不修改 trading_state”与“不用 LLM 乱猜主力”的五大禁止事项。所有指标均基于物理 Tick 差值严谨计算。
